# Bedrock Agent Call Summary Format Documentation

## Overview

This document describes the differences between call summary JSON formats sent by regular SignalWire agents and Bedrock agents, identifies the causes of UI crashes, and provides the correct format that Bedrock agents should use.

## Current Issues

### 1. Missing Numeric Fields

Bedrock agents are missing critical numeric fields that the UI expects for displaying call statistics and metrics:

| Field Name | Description | Expected Type | UI Usage |
|------------|-------------|---------------|----------|
| `total_minutes` | Total call duration in minutes | number | Duration display |
| `total_input_tokens` | Total tokens sent to the AI model | integer | Token usage stats |
| `total_output_tokens` | Total tokens generated by the AI model | integer | Token usage stats |
| `total_asr_minutes` | Total ASR (speech recognition) minutes | number | ASR usage metrics |
| `total_asr_cost_factor` | ASR cost multiplier | number | Cost calculations |
| `total_tts_chars` | Total TTS characters generated | integer | TTS usage metrics |
| `total_tts_chars_per_min` | TTS characters per minute rate | number | TTS rate display |
| `total_wire_input_tokens` | Wire protocol input tokens | integer | Advanced metrics |
| `total_wire_input_tokens_per_minute` | Wire input token rate | number | Rate calculations |
| `total_wire_output_tokens` | Wire protocol output tokens | integer | Advanced metrics |
| `total_wire_output_tokens_per_minute` | Wire output token rate | number | Rate calculations |
| `ai_start_date` | AI session start timestamp | integer | Session timing |
| `ai_end_date` | AI session end timestamp | integer | Session timing |
| `times` | Array of response timing data | array | Performance metrics |

### 2. Timestamp Format Inconsistency

- **Regular agents**: Use integer timestamps (Unix microseconds)
- **Bedrock agents**: Use ISO 8601 string timestamps

This causes type mismatches when the UI tries to parse and format timestamps.

### 3. Missing Latency Fields in Call Log

Regular agents include performance metrics in assistant responses:
- `latency`: Processing time in milliseconds
- `audio_latency`: Time to generate audio in milliseconds  
- `utterance_latency`: Time to first spoken word in milliseconds

Bedrock agents are missing these fields entirely, which prevents performance analysis.

### 4. Missing SWMLVars Fields

Regular agents include `ai_result` in the `SWMLVars` object, while Bedrock agents only have `answer_result`.

### 5. UI Crash Points

The UI crashes occur in two main components:

#### CallSummaryDetail.tsx
- Lines attempting to call `.toFixed()` on undefined values
- Expects numeric values for rate calculations

#### CallSummariesList.tsx  
- Lines attempting to call `.toLocaleString()` on undefined token counts
- Expects integer values for formatting

## Field Format Specifications

### Timestamps
- **Format**: Unix timestamp in microseconds (multiply seconds by 1,000,000)
- **Type**: Integer
- **Example**: `1752553551987564` represents `2025-07-15T04:25:51.987564Z`

### Call Log Entries
Each entry in `call_log` and `raw_call_log` must have:
- `role`: String - One of: "system", "user", "assistant", "function"
- `content`: String - The message content
- `timestamp`: Integer - Unix microseconds timestamp

Additionally, **assistant** role entries should include latency metrics:
- `latency`: Integer - Processing latency in milliseconds
- `audio_latency`: Integer - Audio generation latency in milliseconds
- `utterance_latency`: Integer - Time to first utterance in milliseconds

**User** role entries may include:
- `confidence`: Number - ASR confidence score (0.0 to 1.0)

### SWAIG Log Entries
Each entry in `swaig_log` represents a function call:
- `action`: String - The function name
- `args`: Object - Function arguments
- `result`: Object - Function execution result
- `timestamp`: Integer - Unix microseconds timestamp

### Times Array
Performance metrics for each AI response:
- `tps`: Number - Tokens per second
- `tokens`: Integer - Number of tokens in response
- `avg_tps`: Number - Average tokens per second
- `response`: String - The actual response text
- `token_time`: Number - Time to generate tokens (seconds)
- `answer_time`: Number - Total response time (seconds)
- `response_word_count`: Integer - Number of words in response

### Numeric Fields
All numeric fields should be included even if zero:
- Duration fields (`total_minutes`, etc.): Use 0 if not applicable
- Token counts: Use 0 if not applicable
- Rate fields: Calculate as `value / total_minutes` or use 0

## Migration Checklist

To update Bedrock agents to use the correct format:

1. ✅ Add all missing numeric fields with appropriate values
2. ✅ Convert timestamp strings to integer microseconds
3. ✅ Include `times` array with response performance data
4. ✅ Add `ai_start_date` and `ai_end_date` timestamps
5. ✅ Include token usage metrics (even if estimated)
6. ✅ Add `ai_result` to `SWMLVars`
7. ✅ Ensure all rate fields are calculated and included

## Notes

- The UI has been patched to handle missing fields gracefully, but proper data structure alignment is the recommended solution
- All timestamps should use the same format (Unix microseconds) for consistency
- Missing fields should be included with sensible defaults rather than omitted
- The `conversation_summary` field unique to Bedrock can remain as an additional field
